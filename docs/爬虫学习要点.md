真实问题和解决思路整理成可复用的知识点。面向刚入门的爬虫学习者，重在“为什么这么做”，
而不只是“怎么做”。

配套代码：`login.py`（自动登录）、`use_session.py`（复用会话 + 订购 + 取订阅链接）。

## 1. 先搞清楚：这是什么类型的网站

爬虫的第一步不是写代码，是**判断网站类型**，它决定了整套打法：

- **传统服务端渲染（SSR）**：HTML 里直接有数据，`requests` + 解析库就够了。
- **单页应用（SPA，本项目就是）**：服务器只返回一个空壳 `<div id="app">`，数据靠
  JS 异步请求接口再渲染。`requests` 抓回来是空的，必须用**能跑 JS 的浏览器**
  （Playwright / Selenium）或者**直接调它的接口**。

> 判断方法：右键“查看网页源代码”，如果看不到页面上的文字/数据，只有一堆 `<script>`，
> 基本就是 SPA。

本项目用的是 **Playwright**：它能启动真实的 Chromium，像真人一样点击、输入、等待。

## 2. 要点一：真实事件 vs 合成事件（isTrusted）

**遇到的问题**：一开始想“图快”，用 JS 直接给输入框赋值：
```js
input.value = "账号"   // ❌ 这样填
```
结果表单被网站重置、账号框被清空，填了个寂寞。

**原因**：浏览器给每个事件打了一个 `isTrusted` 标记。
- 真人点击/打字 → `isTrusted: true`
- JS 合成的 `input`/`change` 事件 → `isTrusted: false`

很多前端框架（本项目是 Vuetify/Vue）会**检测这个标记**，判定 `false` 的是非法输入，
直接丢弃或重置表单。

**解决**：用 Playwright 的 `click()` / `type()`。它们走的是**浏览器层面的真实输入**，
和真人一模一样，`isTrusted: true`，自然通过。

```python
def fill_like_human(page, selector, value):
    field = page.locator(selector)
    field.click()              # 真实点击获得焦点
    field.fill("")             # 先清空
    field.type(value, delay=40)  # 逐字键入，delay 模拟真人打字节奏
```

> **学习点**：能用工具的“真实输入”就别用 JS 注入 value。`delay` 参数让节奏更像人，
> 也能降低被行为检测识别的概率。

## 3. 要点二：用 Googlebot UA 绕过“开发者工具检测”导致的页面刷新

**遇到的问题**：开着窗口（非 headless）跑脚本时，页面**疯狂刷新 / 被弹到 /error-page**，
根本没法操作。

**排查过程**（这才是重点）：
1. 观察到只有“有窗口”时才刷新，headless 反而正常 → 怀疑是**反调试/开发者工具检测**。
2. 查页面加载的 JS，发现用了开源库 **`disable-devtool`**（专门检测 F12 是否打开）。
3. 读这个库的源码，发现它有一条 **SEO 白名单**：
   ```js
   if (seo && seoBot) return "seobot"   // 命中搜索引擎爬虫 → 直接放行，不启动检测
   ```
   也就是说，当它认为访问者是**搜索引擎爬虫**时，会主动关掉检测——因为网站希望被
   Google 收录，不能把爬虫也弹走。

**解决**：把浏览器的 User-Agent 伪装成 **Googlebot**，命中这条白名单，检测器根本不启动。
```python
context = browser.new_context(
    user_agent="Mozilla/5.0 (compatible; Googlebot/2.1; "
               "+http://www.google.com/bot.html)"
)
```
这样无论 headless 还是有窗口，都不再刷新。

> **学习点**：
> - 反爬/反调试大多是**开源库或常见套路**，遇到诡异行为先找“它用了什么库”，去读源码找它的“后门/白名单”，往往比硬刚更省事。
> - 网站为了 SEO 通常会给搜索引擎爬虫开绿灯，**Googlebot UA 是一个通用的低成本尝试**。
> - 这比“拦截跳转”“爆破检测 token”之类的对抗手段都更干净——**从源头让检测不发生，而不是事后补救**。

## 4. 要点三：登录态存在哪里？localStorage 还是 Cookie

**遇到的问题**：登录成功后，想“免登录”复用会话，直接下次带上 Cookie 却进不去。

**原因**：这个站的登录态**不在 Cookie 里**，而是存在 `localStorage`：
- `localStorage.user`（内含 `token`、`auth_data`）
- `localStorage.auth_data`
- API 靠请求头 `Authorization: Bearer <token>` 鉴权
- 前端路由守卫检查 `localStorage` 里有没有 `auth_data`，没有就踢回 `/login`

**解决**：
1. 登录成功后，把这两个 key 从 `localStorage` 导出存成文件：
   ```python
   auth = page.evaluate(
       "() => ({ user: localStorage.getItem('user'),"
       " auth_data: localStorage.getItem('auth_data') })"
   )
   ```
2. 下次复用时，用 **`add_init_script`** 在**页面任何 JS 运行之前**把它们写回 `localStorage`：
   ```python
   context.add_init_script(init_script)  # 在每个新文档最早期执行
   page = context.new_page()
   page.goto(target_url)
   ```
   关键在“**最早期**”——路由守卫一加载就会读 `localStorage`，token 必须在那之前就位，
   否则一进去就被踢回登录页。

> **学习点**：
> - 会话不一定在 Cookie 里，现代 SPA 常用 `localStorage` / `sessionStorage`。用浏览器
>   开发者工具的 Application 面板确认到底存在哪。
> - `add_init_script` 是复用登录态的利器：它保证脚本在页面自己的 JS **之前**运行。
> - Cookie 有 Playwright 的 `storage_state` 可以整体保存/恢复；纯 localStorage 方案则
>   需要像这样手动导出+注入。

## 5. 要点四：SPA 是异步的——等待策略

**遇到的问题**：`goto` 之后马上找元素，报“找不到”；或者刚进页面数据还没加载。

**原因**：SPA 的渲染、路由跳转、数据请求全是**异步**的。`domcontentloaded` 只表示
HTML 骨架到了，不代表数据/组件已经渲染好。

**解决**：用**条件等待**，而不是死等固定时间。
```python
# 等某个元素出现（最多等 15 秒）
page.wait_for_selector("input[type=password]", timeout=15000)

# 等 URL 变成期望的样子（比如登录成功后跳转）
page.wait_for_url(lambda u: "/login" not in u, timeout=15000)

# 轮询某个条件直到满足（本项目等 localStorage 里出现订阅链接）
for _ in range(24):
    v = page.evaluate("() => localStorage.getItem('subscribe_url')")
    if v and v.startswith("http"):
        break
    page.wait_for_timeout(500)
```

> **学习点**：
> - **优先用条件等待**（`wait_for_selector` / `wait_for_url`），只在没有明确信号时才用
>   固定 `wait_for_timeout` 兜底。
> - 异步数据没到位就读，会拿到空值或**占位值**（本项目里就踩过 `default-url` 的坑，见下节）。
> - 给等待设**超时**并处理超时分支，别让脚本无限卡住。

## 6. 要点五：拿数据的思路演进（剪贴板 → 网络接口 → 前端内存）

这是本项目**最有价值的一段思考**：同一个“获取订阅链接”的需求，方案一步步进化。

### 背景
页面有个“复制通用订阅链接”按钮，但链接**不在**任何 `<input>` / `href` / `data-*` 属性里
（探测确认过），点一下才会把链接放进剪贴板。

### 方案①：模拟点击 + 读剪贴板（能用，但不优雅）
```python
context = browser.new_context(permissions=["clipboard-read", "clipboard-write"])
# 点按钮 → 读 navigator.clipboard.readText()
```
**踩的坑**：链接是异步生成的，刚进页面就点，剪贴板里是占位值 `default-url`，
而且**剪贴板不会随数据更新自动刷新**。只能“点击+读取”循环重试，直到读到 `http` 开头。
**缺点**：依赖剪贴板权限、要处理占位值、逻辑绕。

### 方案②：直接调它的接口（更直接，但此站行不通）
挂网络监听，发现链接来自接口 `GET /api/v1/user/getSubscribe`（v2board/Xboard 家族）。
理论上带着 token 直接请求这个接口就行，连浏览器都不用开。
**踩的坑**：这个站的**API 响应是加密的**，直接调接口只能拿到一坨密文
（`bCJrbzVv...`），要还原得逆向前端的解密算法——成本太高，不划算。

### 方案③：从前端解密后的内存/存储读取（最干净）✅
既然剪贴板能拿到明文，说明前端**一定在某处存了解密后的链接**。再探测一次，发现：
> 进入面板页后，前端会自行请求接口、解密，并把结果写入 **`localStorage.subscribe_url`**。

于是直接读它就行——不点按钮、不碰剪贴板、不处理占位值：
```python
for _ in range(24):
    v = page.evaluate("() => localStorage.getItem('subscribe_url')")
    if v and v.startswith("http"):
        print(v); break
    page.wait_for_timeout(500)
```

### 思路总结（重要）
获取数据时，按“**离原始数据越近越好、但要避开加密/对抗**”排优先级：

| 优先级 | 方案 | 适用场景 | 本项目 |
|---|---|---|---|
| 高 | 直接调 API | 接口无加密、鉴权可复现 | ❌ 响应加密 |
| 高 | 读前端解密后的 localStorage / 内存 / DOM | 前端已经解密并存下了 | ✅ 最终选它 |
| 中 | 模拟操作 UI + 读结果（剪贴板/弹窗） | 前面都不行时的兜底 | 一开始用的 |

> **学习点**：
> - **别一条路走到黑**。第一个能跑的方案（剪贴板）往往不是最好的。
> - 遇到“响应加密”，先别急着逆向解密算法——**前端最终一定要用明文**，去它解密后的
>   落脚点（localStorage / Vue 响应式数据 / DOM）读，通常能白嫖它的解密结果。
> - 网站会**动态分配域名**（本项目订阅域名多次变化），所以更不能硬编码 URL，要读实时值。

## 7. 要点六：探测优先——先看清楚再动手

本项目每加一个功能，都先写一个 `_probe_xxx.py` 临时脚本去“看清楚页面”，再写正式代码。
探测脚本干的事：
- dump 所有按钮的文字和 class（找“立即订购”“复制”这些按钮的选择器）
- dump 所有 `input` 的 placeholder（找优惠码输入框）
- 检测有没有弹窗/遮罩层挡住点击
- 挂网络监听，看点某个按钮触发了哪些接口、返回什么
- 扫描 DOM / localStorage / Vue 组件数据，定位某个值存在哪

```python
# 例：dump 页面上所有按钮
btns = page.evaluate("""() => Array.from(document.querySelectorAll('button,.v-btn'))
    .map(b => ({text: b.innerText.trim(), cls: b.className, disabled: b.disabled}))
    .filter(b => b.text)""")
```

> **学习点**：
> - **`page.evaluate` 是探测神器**：能在页面上下文里跑任意 JS，把你想看的信息捞出来。
> - 探测脚本用 `_` 前缀命名（如 `_probe_plan.py`），用完即删，**不提交进仓库**。
> - “选择器怎么写才稳”不靠猜，靠探测：本项目发现“每月付款”是个 `<h5>`，真正可点击的是
>   它的祖先 `.price-option-row`，用文本过滤定位整行才可靠。

## 8. 要点七：安全阀与工程习惯

爬虫/自动化会**真的操作线上系统**，容易误触不可逆动作（下单、删除、发消息）。

**本项目的做法**：订购流程默认走到“立即订购”**前就停下**，只把表单准备好，
必须显式加 `--confirm` 才真正点击下单：
```python
if confirm:
    order.click()          # 真正提交
else:
    print("表单已就绪，安全阀生效：未真正下单。加 --confirm 才提交。")
```
测试选择器/优惠码时用假数据、不加 `--confirm`，就不会产生真实订单。

**其他工程习惯**：
- **凭证隔离**：账号密码放 `credentials.py`，会话 token 放 `session_state.json`，
  两者都写进 `.gitignore`，**绝不提交**。仓库里只放 `credentials.example.py` 模板。
- **参数化**：功能用命令行开关控制（`--order` / `--sub` / `--coupon` / `--confirm`），
  互不干扰，默认行为保守（只验证会话）。
- **临时探测脚本不进仓库**，正式逻辑才提交。

> **学习点**：面对不可逆操作，**默认不做、显式确认才做**。这是自动化脚本的基本安全素养。

## 9. 速查表

| 场景 | 手段 |
|---|---|
| 网站是 SPA，requests 抓不到数据 | 用 Playwright 跑真实浏览器 |
| JS 注入 value 被表单重置 | 用 `click()` + `type(delay=…)` 走真实事件（isTrusted:true） |
| 有窗口就疯狂刷新 / 被弹走 | 查是不是 `disable-devtool` 类库，用 Googlebot UA 命中 SEO 白名单 |
| 免登录复用会话 | 找到登录态存哪（localStorage/Cookie），用 `add_init_script` 最早期注入 |
| 元素找不到 / 数据是空的 | 用 `wait_for_selector` / `wait_for_url` 条件等待，别死等 |
| 拿到的是占位值 | 数据异步未就位，轮询到真值再用 |
| 要拿某个值，UI 上不好取 | 优先：调 API → 读 localStorage/内存 → 兜底模拟 UI |
| 接口响应是加密的 | 别逆向解密，去前端解密后的落脚点（localStorage/DOM）读明文 |
| 不知道选择器怎么写 | 写 `_probe` 脚本 + `page.evaluate` dump 页面结构 |
| 有不可逆操作（下单等） | 加 `--confirm` 安全阀，默认不执行 |
| 账号/token 等敏感信息 | 放进 `.gitignore`，仓库只留 `.example` 模板 |
